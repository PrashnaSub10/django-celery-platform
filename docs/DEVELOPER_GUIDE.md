# Developer Guide — Wiring Your Django Project to the Platform

This guide walks you through connecting your Django project to the
composable Celery platform.  Follow the steps in order on a first
integration; return to individual sections when changing broker strategy
or enabling new features.

---

## The `config/` package

`components/workers/config/` is the canonical reference implementation.
Copy the files you need into your Django project's own `config/` directory.

| File | Purpose | When to use |
|---|---|---|
| `broker_settings.py` | URL builders + immutable conf dicts | Always — every other module imports this |
| `celery_config.py` | Standalone apps for **host-side** dispatch | Your Django app runs on the host, not in a container |
| `celery_hybrid.py` | Dual-broker apps for **containerised** workers | Workers run inside Docker on `celery-broker-net` |
| `django_celery_integration.py` | Django-aware apps with `autodiscover_tasks` | Full Django integration with `@shared_task` support |
| `path_utils.py` | Cross-platform, traversal-safe path helpers | Any task that reads/writes files |

### Why `broker_settings.py` exists

All broker URLs and Celery conf dicts live in one place.  Every other
module imports from it — nothing is duplicated.  The conf dicts are
`MappingProxyType` (immutable), so a caller doing
`REDIS_CONF["key"] = val` raises `TypeError` immediately rather than
silently mutating shared state.

### The `_make_apps()` factory pattern

Every config module wraps Celery instantiation in a `_make_apps()`
function rather than creating apps at module level:

```python
def _make_apps() -> tuple[Celery, Celery]:
    ...

app_redis, app_rabbitmq = _make_apps()
```

This matters because `broker_settings._require_env()` raises
`RuntimeError` when a required env var is absent.  If apps were
instantiated at module level, **any import** of the module without the
environment configured would crash the entire process — including
management commands, test runners, and IDE introspection.  The factory
defers the env read to the moment the module is first imported with a
live environment.

## Step 0 — ASGI / WebSocket (optional)

Skip this step if your project does not use WebSockets or Django Channels.

### Install dependencies

```bash
pip install daphne channels channels-redis
```

### Create `config/asgi.py`

```python
import os
from django.core.asgi import get_asgi_application
from channels.routing import ProtocolTypeRouter, URLRouter
from channels.auth import AuthMiddlewareStack
import myapp.routing

os.environ.setdefault("DJANGO_SETTINGS_MODULE", "config.settings.production")

application = ProtocolTypeRouter({
    "http": get_asgi_application(),
    "websocket": AuthMiddlewareStack(
        URLRouter(myapp.routing.websocket_urlpatterns)
    ),
})
```

### Add Channel Layer to Django settings

`docker-compose.asgi.yml` injects `CHANNELS_REDIS_HOST` and
`CHANNELS_REDIS_PORT` into the Daphne container automatically.
Use those env vars so the same settings file works in both Docker
and local development:

```python
CHANNEL_LAYERS = {
    "default": {
        "BACKEND": "channels_redis.core.RedisChannelLayer",
        "CONFIG": {
            "hosts": [{
                "host": os.environ.get("CHANNELS_REDIS_HOST", "localhost"),
                "port": int(os.environ.get("CHANNELS_REDIS_PORT", 6380)),
                "password": os.environ["CHANNELS_REDIS_PASSWORD"],
            }],
        },
    },
}
```

`CHANNELS_REDIS_PASSWORD` is auto-generated by `init-secrets.sh` and
is separate from `REDIS_PASSWORD` — the Channel Layer Redis uses
`noeviction` policy while the Celery broker Redis uses `allkeys-lru`.
They must not share an instance.

### Launch with ASGI enabled

```bash
ASGI_MODE=true \
ASGI_APPLICATION=config.asgi:application \
MODE=full BROKER_MODE=hybrid SERVER_PROFILE=large \
  PROJECT_PROFILE=my-project.env ./core/up.sh
```

`ASGI_APPLICATION` tells the Daphne container which Python dotted path
to load.  It defaults to `config.asgi:application` if not set.

Nginx routes `/ws/` to Daphne on port 9501.  All other traffic goes to
your WSGI app on port 9845.

### Validate

```bash
# npm install -g wscat
wscat -c wss://yourdomain.com/ws/test/ --no-check
# Expected: HTTP 101 Switching Protocols
```

## Step 1 — Launch the platform

```bash
cd django-celery-platform

# Standard monitoring, Redis broker, medium sizing
MODE=standard BROKER_MODE=redis SERVER_PROFILE=medium \
  PROJECT_PROFILE=/path/to/my/celery-profile.env \
  ./core/up.sh

# Full mode, hybrid brokers, dual workers, large sizing
MODE=full BROKER_MODE=hybrid WORKER_MODE=dual SERVER_PROFILE=large \
  PROJECT_PROFILE=/path/to/my/celery-profile.env \
  ./core/up.sh
```

Valid values:

| Variable | Options |
|---|---|
| `MODE` | `minimal` · `standard` · `full` |
| `BROKER_MODE` | `redis` · `rabbitmq` · `hybrid` |
| `WORKER_MODE` | `single` (default) · `dual` |
| `SERVER_PROFILE` | `small` · `medium` · `large` |

`up.sh` validates all values against an allowlist before constructing
any file paths. `WORKER_MODE=dual` requires `BROKER_MODE=hybrid` —
`up.sh` exits with an error if this constraint is violated.

## Step 1.5 — Choose your code delivery method (`CODE_SOURCE`)

The platform is project-agnostic: it does not care how your Django app is deployed.
Set `CODE_SOURCE` in your `celery-profile.env` to tell the workers how to find your code.

| `CODE_SOURCE` | Use when | Required variable |
|---|---|---|
| `bind` (default) | Django runs as a systemd service or directly on the host | `APP_PATH` |
| `image` | You maintain a custom worker image with your code baked in | — |
| `volume` | Django is containerised and shares a Docker volume with workers | `APP_VOLUME_NAME` |
| `git` | Cloud/CI deployments — workers clone the repo at startup | `APP_GIT_URL` |
| `pip` | Your Django project is distributed as a PyPI package | `APP_PIP_PACKAGE` |

---

### `CODE_SOURCE=bind` — Django on the Docker host (default)

Your Django project directory is mounted directly into the worker containers.
Any change to the source on the host is reflected immediately without rebuilding.

```env
CODE_SOURCE=bind
APP_PATH=/home/deploy/my-django-project
```

**Requirement:** The `celery` user inside the container (non-root) needs **read** access
to `APP_PATH`. Ensure the directory has group-read or world-read permissions:

```bash
chmod o+rX -R /home/deploy/my-django-project
```

---

### `CODE_SOURCE=image` — Custom worker image (code baked in)

Build a custom Docker image that `COPY`s your project code in.
No `APP_PATH`, `APP_VOLUME_NAME`, or `APP_GIT_URL` is required.

```dockerfile
# Dockerfile.myproject — extends the base worker image
FROM celery-microservice:base

# Copy your project code into the image
COPY --chown=celery:celery . /app

# Install project-specific requirements
COPY requirements_celery.txt /tmp/
RUN pip install --no-cache-dir -r /tmp/requirements_celery.txt
```

```bash
docker build -t celery-microservice:myproject \
  -f Dockerfile.myproject /path/to/my-django-project
```

```env
CODE_SOURCE=image
WORKER_IMAGE=celery-microservice:myproject
```

Use this mode in production when you want fully reproducible, immutable worker images —
the same image that passed CI/CD is exactly what runs in production.

---

### `CODE_SOURCE=volume` — Containerised Django sharing a volume

Use when your Django application runs in Docker and both the Django container
and the Celery workers can mount the same named volume.

**In your Django app's `docker-compose.yml`:**

```yaml
services:
  django:
    image: my-django-app
    volumes:
      - app_code:/app

volumes:
  app_code:
```

**In `celery-profile.env`:**

```env
CODE_SOURCE=volume
APP_VOLUME_NAME=app_code
```

`up.sh` declares `app_code` as an external volume (created outside the platform's
compose stack). The `celery` user inside the worker needs read access to the volume's
contents — ensure file ownership is compatible.

---

### `CODE_SOURCE=git` — Clone from a git repository

The worker's `docker-entrypoint.sh` runs `git clone` on first start and
`git pull --ff-only` on subsequent starts (the clone persists in a named volume
called `git_code`).

```env
CODE_SOURCE=git
APP_GIT_URL=https://github.com/your-org/your-django-project.git
APP_GIT_BRANCH=main    # optional; defaults to repo default branch
```

**Private repositories** — inject credentials via the URL (do not commit tokens):

```env
# GitHub personal access token
APP_GIT_URL=https://x-access-token:${GITHUB_TOKEN}@github.com/your-org/repo.git

# GitLab deploy token
APP_GIT_URL=https://deploy-token:${GITLAB_TOKEN}@gitlab.com/your-org/repo.git
```

`GITHUB_TOKEN` / `GITLAB_TOKEN` can be set in `.env.secrets` alongside the other secrets.

> **Note:** On each `docker compose up` (not restart), workers will `git pull` to sync
> with the latest commit. To force a fresh clone (e.g. after a branch change), remove the
> `git_code` volume: `docker volume rm <project>_git_code`.

---

### `CODE_SOURCE=pip` — Install from PyPI

The worker's `docker-entrypoint.sh` runs `pip install APP_PIP_PACKAGE` at startup.
Your Django project must be packaged and published (PyPI, private registry, or VCS URL).

```env
CODE_SOURCE=pip
APP_PIP_PACKAGE=your-django-project==1.2.3
```

VCS or local path installs also work:

```env
APP_PIP_PACKAGE=git+https://github.com/your-org/repo.git@v1.2.3#egg=yourproject
```

Point `CELERY_APP_REDIS` and `CELERY_APP_RABBITMQ` at the installed package's module:

```env
CELERY_APP_REDIS=yourproject.celery_redis:app
CELERY_APP_RABBITMQ=yourproject.celery_rabbitmq:app
```

Because the code lives in site-packages (not in `/app`), `PYTHONPATH=/app` is still
set but is effectively a no-op. The installed package is found via `sys.path` normally.

> **Note:** `pip install` runs on every container start. For production, prefer
> `CODE_SOURCE=image` with a pinned version baked in — it's faster and reproducible.

## Step 2 — Configure your Django app

Choose the section that matches your `BROKER_MODE`.

---

### `BROKER_MODE=redis` — speed-first

Best for: notifications, cache invalidation, real-time events, log
processing.  Tasks may be lost if Redis restarts under memory pressure.

**Copy `broker_settings.py` into your project, then create
`config/celery_redis.py`:**

```python
import os
from celery import Celery
from config.broker_settings import REDIS_CONF, redis_broker_url, redis_backend_url
from django.conf import settings as django_settings


def _make_app() -> Celery:
    os.environ.setdefault("DJANGO_SETTINGS_MODULE", "config.settings.production")
    app = Celery("my_project", broker=redis_broker_url(), backend=redis_backend_url())
    app.config_from_object("django.conf:settings", namespace="CELERY")
    app.conf.update(REDIS_CONF)
    app.autodiscover_tasks(lambda: django_settings.INSTALLED_APPS)
    return app


app = _make_app()
```

**`config/__init__.py`:**

```python
from .celery_redis import app

__all__ = ("app",)
```

---

### `BROKER_MODE=rabbitmq` — reliability-first

Best for: payments, financial transactions, emails, audit logs.  Tasks
survive a worker crash (`task_acks_late=True`, `task_reject_on_worker_lost=True`).

**Create `config/celery_rabbitmq.py`:**

```python
import os
from celery import Celery
from config.broker_settings import RABBITMQ_CONF, rabbitmq_broker_url
from django.conf import settings as django_settings


def _make_app() -> Celery:
    os.environ.setdefault("DJANGO_SETTINGS_MODULE", "config.settings.production")
    app = Celery("my_project", broker=rabbitmq_broker_url(), backend="rpc://")
    app.config_from_object("django.conf:settings", namespace="CELERY")
    app.conf.update(RABBITMQ_CONF)
    app.autodiscover_tasks(lambda: django_settings.INSTALLED_APPS)
    return app


app = _make_app()
```

---

### `BROKER_MODE=hybrid` + `WORKER_MODE=single` — hybrid brokers, independent monitoring

Both `worker-fast` and `worker-critical` run. Each has its own Flower
instance (`:5555` for Redis workers, `:5556` for RabbitMQ workers).
Use this when you want hybrid brokers but prefer to monitor each pool
independently or when resource constraints make a third Flower instance
undesirable.

**Copy `celery_hybrid.py` from `components/workers/config/` into your
project's `config/` directory unchanged.** The module reads `REDIS_HOST`
and `RABBITMQ_HOST` from the environment at import time, so it works in
both Docker (container hostnames) and local development (localhost)
without modification:

```bash
# Docker — resolved via celery-broker-net
REDIS_HOST=celery-redis-shared
RABBITMQ_HOST=celery-rabbitmq-shared

# Local development — override to reach published ports
REDIS_HOST=localhost
RABBITMQ_HOST=localhost
```

**Dispatch from your views or tasks:**

```python
from config.celery_hybrid import app_redis, app_rabbitmq

# Fast lane — ephemeral, high-throughput
app_redis.send_task("tasks.send_notification", args=[user_id])

# Secure vault — durable, guaranteed delivery
app_rabbitmq.send_task("tasks.process_payment", args=[order_id])
```

**`config/__init__.py`:**

```python
from .celery_hybrid import app_redis, app_rabbitmq

__all__ = ("app_redis", "app_rabbitmq")
```

---

### `BROKER_MODE=hybrid` + `WORKER_MODE=dual` — unified monitoring

Same worker pair as `single`, but adds two services from
`docker-compose.dual-workers.yml`:

- `flower-hybrid` at `:5557` — a single Flower UI showing `fast@` and
  `critical@` workers side by side with unified task history and failure
  rates across both brokers
- `worker-hybrid-beat` — a single Beat scheduler that dispatches
  periodic tasks to either broker queue; the per-broker `celery-beat`
  from `docker-compose.workers.yml` is scaled to 0 automatically to
  prevent duplicate task execution

The Django-side code is identical to `single` — same `celery_hybrid.py`,
same dispatch pattern. The difference is purely in the infrastructure
layer.

**`celery-profile.env` — both entrypoints required:**

```env
CELERY_APP_REDIS=config.celery_hybrid:app_redis
CELERY_APP_RABBITMQ=config.celery_hybrid:app_rabbitmq
CELERY_REDIS_QUEUE=redis_queue
CELERY_RABBITMQ_QUEUE=rabbitmq_queue
```

**Verify dual mode is healthy:**

```bash
# Unified Flower — both worker pools visible at one URL
open http://localhost:5557

# Confirm both worker hostnames respond
docker exec <project>-worker-fast \
  python3 -m celery -A config.celery_hybrid:app_redis inspect ping
# Expected: fast@<hostname> responds

docker exec <project>-worker-critical \
  python3 -m celery -A config.celery_hybrid:app_rabbitmq inspect ping
# Expected: critical@<hostname> responds

# Confirm only one Beat process is running
docker ps --filter name=beat
# Expected: only <project>-beat-hybrid, NOT <project>-beat
```

---

### Full Django integration (`@shared_task`)

If you use `@shared_task` decorators and want both apps to
autodiscover tasks across all installed apps, copy
`django_celery_integration.py` as your `config/celery.py`:

```python
# config/celery.py  ←  copy of django_celery_integration.py
# Both app_redis and app_rabbitmq are wired to Django settings
# and call autodiscover_tasks lazily to avoid circular imports.
```

```python
# config/__init__.py
from .celery import app_redis, app_rabbitmq

__all__ = ("app_redis", "app_rabbitmq")
```

The `config_from_object` → `conf.update` ordering is intentional:
platform defaults in `broker_settings.py` are applied **after**
Django settings so they are not silently overridden by an absent or
partial `CELERY_*` block in your settings file.

## Step 3 — Create your `celery-profile.env`

This file is the contract between your Django project and the
`docker-compose.workers.yml` template. Set `CODE_SOURCE` here to match
how your Django project is deployed — see Step 1.5 above for all options.

```env
# ── Required: identity ────────────────────────────────────────
PROJECT_NAME=my-app
WORKER_IMAGE=celery-microservice:base

# ── CODE_SOURCE: how your project code reaches the workers ────
# Options: bind | image | git | volume | pip  (see Step 1.5)
CODE_SOURCE=bind
APP_PATH=/absolute/path/to/your/django-project   # required for CODE_SOURCE=bind
# APP_VOLUME_NAME=my_app_code                    # required for CODE_SOURCE=volume
# APP_GIT_URL=https://github.com/your-org/repo   # required for CODE_SOURCE=git
# APP_GIT_BRANCH=main                            # optional for CODE_SOURCE=git
# APP_PIP_PACKAGE=your-django-project==1.2.3     # required for CODE_SOURCE=pip

# ── Celery app entrypoints ────────────────────────────────────
# Point to the module:attribute created in Step 2.
CELERY_APP_REDIS=config.celery_redis:app
CELERY_APP_RABBITMQ=config.celery_rabbitmq:app   # hybrid/rabbitmq only

# ── Queue names ───────────────────────────────────────────────
# Must match task_default_queue in broker_settings.py (or your override).
CELERY_REDIS_QUEUE=redis_queue
CELERY_RABBITMQ_QUEUE=rabbitmq_queue

# ── Django ────────────────────────────────────────────────────
DJANGO_SETTINGS_MODULE=config.settings.production
DJANGO_ALLOWED_HOSTS=yourdomain.com

# ── Worker image selection ────────────────────────────────────
# Use a capability image if your tasks need extra system libraries:
#   celery-microservice:mssql   — SQL Server / pyodbc
#   celery-microservice:pdf     — PDF generation / image processing
#   celery-microservice:smb     — SMB/SFTP file shares
#   celery-microservice:full    — all of the above (legacy only)
# WORKER_IMAGE=celery-microservice:mssql

# ── Runtime pip injection (development only) ──────────────────
# EXTRA_PIP_PACKAGES requires ALLOW_RUNTIME_PIP=true in production.
# In production, prefer CODE_SOURCE=image or CODE_SOURCE=pip.
# EXTRA_PIP_PACKAGES=pysmb==1.2.13 paramiko==3.4.0
# ALLOW_RUNTIME_PIP=false
```

## Step 4 — Using `path_utils` in your tasks

`path_utils.py` provides traversal-safe path resolution (CWE-22
hardened) for tasks that read or write files.  Copy it into your
project's `config/` directory alongside `broker_settings.py`.

```python
from config.path_utils import get_log_path, get_data_path, ensure_dir

@app.task
def export_report(report_id: int) -> str:
    out_dir = get_data_path("reports")
    ensure_dir(out_dir)
    out_file = out_dir / f"{report_id}.csv"
    # ... write file ...
    return str(out_file)
```

`_safe_join` rejects absolute subpaths before the join (preventing the
`Path.__truediv__` silent-replacement attack) and verifies containment
after resolution with `strict=False` so the directory need not exist yet.

Available helpers:

| Function | Returns |
|---|---|
| `get_log_path(log_type)` | `/var/log/app/{log_type}` (Linux) · `C:/Logs/{log_type}` (Windows) |
| `get_data_path(data_type)` | `/var/data/app/{data_type}` |
| `get_temp_path(temp_type)` | `/tmp/app/{temp_type}` |
| `get_project_root()` | Directory containing the calling file |
| `get_relative_path(*parts)` | Project-root-relative path, traversal-safe |
| `ensure_dir(path)` | Creates the directory and returns the `Path` |

## Troubleshooting

### `RuntimeError: Required environment variable 'REDIS_PASSWORD' is not set`

The `_make_apps()` factory ran without `.env.secrets` sourced.

```bash
# Verify the secret exists
grep REDIS_PASSWORD .env.secrets

# Re-run init if missing
./init-secrets.sh
```

### Worker starts but shows offline in Flower

The worker connected to a different broker than Flower is monitoring.

```bash
# Check which broker URL the worker resolved
docker exec <project>-worker-fast env | grep REDIS_HOST
# Must be: REDIS_HOST=celery-redis-shared

# Check CELERY_APP_REDIS points to the right module:attribute
docker exec <project>-worker-fast \
  python3 -c "from config.celery_hybrid import app_redis; print(app_redis.conf.broker_url)"
```

### `ImportError` when the worker starts

`PYTHONPATH` is not set or `APP_PATH` is wrong.

```bash
# Confirm the app volume is mounted correctly
docker exec <project>-worker-fast ls /app/config/

# Confirm PYTHONPATH includes /app
docker exec <project>-worker-fast env | grep PYTHONPATH
# Must be: PYTHONPATH=/app
```

### Tasks dispatched but never consumed

Queue name mismatch between the producer and the worker.

```bash
# Check what queue the worker is consuming
docker logs <project>-worker-fast | grep "celery@"
# Look for: -Q redis_queue

# Check what queue the task was sent to
docker exec <project>-flower-redis \
  python3 -m celery -A config.celery_hybrid:app_redis inspect active_queues
```

`CELERY_REDIS_QUEUE` in your `celery-profile.env` must match
`task_default_queue` in `REDIS_CONF` (default: `redis_queue`).

### `ALLOW_RUNTIME_PIP` warning at container start

```
[entrypoint] WARNING: EXTRA_PIP_PACKAGES is set but ALLOW_RUNTIME_PIP is not 'true'.
```

This is intentional.  Runtime pip injection is blocked in production
containers (`CONTAINER_ENV=true`) unless explicitly unlocked.  For
production, build a custom image instead:

```dockerfile
FROM celery-microservice:base
COPY requirements_myproject.txt /tmp/
RUN pip install --no-cache-dir -r /tmp/requirements_myproject.txt
```

For development only:

```env
ALLOW_RUNTIME_PIP=true
EXTRA_PIP_PACKAGES=pysmb==1.2.13
```

### WebSocket connections drop after 60 seconds

Nginx `proxy_read_timeout` defaults to 60s for HTTP locations.  The
`/ws/` location in `nginx.conf.template` sets `proxy_read_timeout 86400s`
explicitly.  If you are seeing drops, confirm the Nginx config was
regenerated after the last gateway restart:

```bash
docker exec celery-nginx-shared nginx -T | grep proxy_read_timeout
# /ws/ block must show: proxy_read_timeout 86400s
```

If the value is missing, the template substitution failed.  Check
`docker logs celery-nginx-shared` for `envsubst` errors.
